基于 Spring Boot 的交易代码托管平台

初始设计

需求

       要求设计一个应用程序，根据客户所给的交易条件实时跟踪价格，接入火币网提供的 api ，实现自动开单平单交易，同时在交易时通知客户。

第一次修改

       由于开单后所委托的交易并不一定马上在交易所全部成交，客户在使用一段时间后要求增加自动撤单功能：

同时客户又增加了新的判断条件。但在测试时发现单纯通过增加 if 判断无法满足客户的预期，因为出现了不同条件间的关联和互斥，各个条件开始出现优先级；同时随着条件的增加，各个条件间的关系开始变得难以分析，原本分别属于三个模块的条件代码在重排后混杂在一起，失去了原本良好的可读性和可分析性，增加新的判断条件变得越来越困难。单线程的设计已经难以满足需求。

第二次修改

       为了恢复各个模块原来的独立性，使用多线程设计，设计三个线程分别用来开单平单和撤单，互斥的订单委托在到达交易所后便会被抵消,从而消除三个模块间的耦合。

重构

需求追加

       客户再次追加新的需求，要求能够可视化控制交易机器人并随时对机器人条件代码进行动态设置；同时为了避免风险，要求机器人所有发出的交易都需要经过客户人为验证，钉钉回复确认后才能将委托提交到交易所。

系统总体设计

       基于 Spring Boot 开发 RESTful APIs 来处理请求与响应，前后端分离。为了实现机器人代码的动态修改，需要使用元编程动态生成代码，编译后在运行时装入 JVM 。由于多个机器人可能同时请求相同的市场数据，而调用火币 api 获取数据需要经过代理，为了节省流量费用，单独设置一个线程来获取公共信息，存入到 Redis 供各机器人调用。由于客户的回复时间不定，而等待回复时仍然需要监视市场信息，利用钉钉提供的回调接口，设置统一回调网关 + MQ 解耦。

机器人代码生成模块设计

       为了方便代码的动态生成，需要先将所有交易机器人的通用类和共用 jar 包整理抽出，打包为一个 Template 模板。在机器人的初始设计阶段，我们已经将模块解耦，用户需要编写的类只有 Buyer、Seller、Monitor，在此基础上还可以进一步封装隐藏，只对用户公开其中的判断函数。生成用户的机器人项目代码仅需要在模板的基础上，在指定目录下动态创建 Buyer.java、Seller.java、Monitor.java ，修改必要的 MQ 订阅频道代码字段，再对项目进行编译（项目代码的动态编译见：动态编译Java项目），最后将 Robot 类加载到 JVM 运行（类的动态加载见：运行时动态加载类）。
       为了防止用户的代码段包含恶意代码（包括创建修改删除文件等系统调用），除了尽可能使三个类与其它工具模块保持独立，还使用正则表达式检查是否含有恶意的系统函数调用。

接口安全性设计

       为了保证接口的安全，采用了以下措施：

  1. 采用 Token 授权认证，防止未授权的用户获取数据，同时设置过期时间
  2. 采用时间戳超时机制，客户端每次请求接口都带上当前的时间戳，服务端收到请求后将时间戳时间与当前时间进行比较，如果时间差大于设定值，返回请求失败
  3. 防止重放攻击，客户端根据 Token 和时间戳生成 sign 签名，服务端对 sign 也进行验证，只有缓存中不存在该 sign 时才接收请求
  4. 未来如果上线，需要采用 HTTPS 协议，敏感数据进行二次加密传输
